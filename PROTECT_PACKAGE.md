# حماية بكج Lingua (Private Package Guide)

هذا الدليل يشرح كيفية تحويل بكج `samehdoush/lingua` إلى بكج خاص (Private) واستخدامه بشكل آمن في مشاريعك الأخرى مثل `ashara-lms`.

## 1. إنشاء مستودع خاص (Private Repository)
الخطوة الأولى هي استضافة الكود على مستودع لا يمكن للعامة الوصول إليه.

1. اذهب إلى [GitHub New Repository](https://github.com/new).
2. اسم المستودع: `lingua`.
3. **الخصوصية (Privacy):** اختر **Private**. هذا الخيار هو المسؤول عن حماية البكج.
4. اضغط **Create repository**.

## 2. رفع الكود (Push Code)
اربط المجلد المحلي بالمستودع الجديد وقم برفع الكود.

```bash
cd c:\Users\sameh\Herd\lingua

# تهيئة Git إذا لم يكن مهيأ (غالباً هو مهيأ لديك)
# git init

# إضافة رابط المستودع (استبدل الرابط برابط مستودعك الجديد)
git remote add origin https://github.com/samehdoush/lingua.git

# رفع الكود
git branch -M main
git push -u origin main

# إنشاء إصدار (Tag) لكي يتعرف عليه Composer
git tag v1.0.0
git push origin v1.0.0
```

## 3. إعداد المصادقة (Authentication)
بما أن المستودع خاص، لن يستطيع Composer تحميله إلا بمفتاح مرور (Token).

### إنشاء Token:
1. اذهب إلى إعدادات حسابك في GitHub: **Settings** > **Developer settings**.
2. اختر **Personal access tokens** > **Tokens (classic)**.
3. اضغط **Generate new token (classic)**.
4. **Note:** اكتب وصفاً، مثلاً `Composer Access`.
5. **Expiration:** اختر مدة الصلاحية (أو No expiration إذا كنت تفضل ذلك).
6. **Scopes (الصلاحيات):** حدد خيار **`repo`** (Full control of private repositories). هذا ضروري جداً.
7. اضغط **Generate token** وانسخ الكود الطويل (يبدأ عادة بـ `ghp_`).

## 4. استخدام البكج في المشاريع (مثل ashara-lms)

### أ. إعداد ملف `composer.json`
في مشروع `ashara-lms`، أخبر Composer أن يبحث عن البكج في مستودعك الخاص.

```json
"repositories": [
    {
        "type": "vcs",
        "url": "https://github.com/samehdoush/lingua.git"
    }
],
"require": {
    "samehdoush/lingua": "^1.0"
}
```

### ب. المصادقة عند التثبيت
عند تشغيل `composer update`، سيطلب منك Composer الـ Token.
- **Username:** اسم مستخدم GitHub الخاص بك (samehdoush).
- **Password:** الصق الـ Token الذي نسخته (وليس كلمة مرور حسابك).

### ج. حفظ المصادقة (اختياري ومستحسن)
لكي لا يطلب منك الـ Token في كل مرة، يمكنك حفظه في ملف `auth.json` بجانب `composer.json` (تأكد من إضافة `auth.json` لملف `.gitignore` لكي لا يرفع للعامة).

أو الأفضل، قم بتخزينه عالمياً في جهازك مرة واحدة:
```bash
composer config --global --auth github-oauth.github.com YOUR_TOKEN_HERE
```

## سؤالك عن حظر الـ IP (IP Allowlist)

**هل يمكن السماح لـ IPs معينة فقط من خلال GitHub؟**

بشكل افتراضي للمستودعات الشخصية (Personal Repositories): **لا**.
GitHub يعتمد على **المصادقة (Authentication)** وليس **الشبكة (Network/IP)** للتحكم في الوصول. طالما الشخص لا يملك الـ Token، لن يستطيع الوصول للكود حتى لو كان معه الرابط.

**متى يتوفر حظر الـ IP؟**
ميزة **GitHub IP Allowlist** تتوفر فقط في خطة **GitHub Enterprise Cloud** للمؤسسات (Organizations). إذا كان حسابك حساب مؤسسة (Organization) ومشترك في خطة Enterprise، يمكنك تفعيل هذا الخيار من إعدادات الأمان للمنظمة، وحينها يمكنك تحديد IPs معينة فقط يمكنها الوصول للمستودعات الخاصة.

**البدائل إذا كنت تحتاج تقييد IP بشدة:**
إذا كان موضوع الـ IP مصيرياً لك ولا تملك GitHub Enterprise، الحل هو عدم استخدام GitHub لاستضافة البكج، بل استضافته على سيرفر خاص بك (Self-hosted GitLab أو Satis Server) وحماية هذا السيرفر بـ Firewall يسمح فقط لـ IPs معينة.

**نصيحتي:**
الاعتماد على **Private Repository + Personal Access Token** هو المعيار العالمي وهو آمن جداً وكافٍ لأغلب الحالات التجارية والشخصية.

## 5. كيف أحمي التوكين عند التسليم للعميل؟ (مشكلة تسريب التوكين)

سؤالك في محله: **"إذا أعطيت العميل ملف `auth.json` فيه التوكين، سيأخذه ويستخدمه في مكان آخر!"**

هنا ننتقل لمستوى أعلى من الحماية، والحلول هي كالتالي:

### الحل الأول: Satis (سيرفر خاص بك - مجاني)
بدلاً من إعطاء العميل التوكين الخاص بحسابك الكامل (الذي يفتح كل مشاريعك)، تقوم بعمل سيرفر وسيط (Repository) خاص بك باستخدام أداة مجانية من Composer اسمها **Satis**.

**كيف تعمل؟**
1. تنصّب Satis على سيرفر خاص بك.
2. Satis يتصل بـ GitHub (باستخدام التوكين الخاص بك مرة واحدة).
3. العميل يتصل بـ Satis الخاص بك وليس بـ GitHub مباشرة.
4. **الميزة الكبرى:** يمكنك إنشاء **اسم مستخدم وكلمة مرور مختلفة لكل عميل** في Satis.
5. إذا توقف عميل عن الدفع أو سرب الحساب، تقوم بحذف حسابه هو فقط من Satis، ويبقى باقي العملاء يعملون.

### الحل الثاني: Private Packagist (مدفوع)
نفس فكرة Satis ولكن كخدمة جاهزة (SaaS). تدفع لهم، وهم يعطونك لوحة تحكم تنشئ منها روابط خاصة لكل عميل (Token لكل عميل). هذا هو الحل الأسهل للشركات.

### الحل الثالث: Deploy Keys (مجاني - لكل مشروع)
إذا كنت لا تريد عمل Satis، وتريد حلاً مجانياً عبر GitHub:
1. في مستودع البكج على GitHub، اذهب إلى **Settings** > **Deploy keys**.
2. أنشئ مفتاح SSH جديد.
3. أعط هذا المفتاح صلاحية **Read Only**.
4. ضع هذا المفتاح في سيرفر العميل.
**العيب:** هذا المفتاح لا يزال يسمح بتحميل البكج، لكنه على الأقل لا يفتح باقي مشاريعك، ويمكنك حذفه لاحقاً لتعطيل وصول هذا العميل تحديداً.

### الحل الرابع: التشفير (IonCube) + نظام ترخيص (Licensing)
حتى لو حمل العميل الكود، الكود مفتوح المصدر (PHP). إذا أردت منعه من تشغيله بدون "رخصة":
1. يجب تشفير الكود (Obfuscation) باستخدام أدوات مثل **IonCube**.
2. تضع كود برمجي داخل البكج يطلب "License Key" ويتصل بسيرفرك للتحقق منه.
3. إذا لم يكن المفتاح صالحاً، يتوقف الكود عن العمل.

**الخلاصة للوضع الحالي:**
للبداية، الحل الأفضل هو **عدم إعطاء العميل التوكين أبداً**.
- إذا كنت أنت من يرفع الموقع للسيرفر: ضع التوكين في السيرفر ولا تعطيه للعميل.
- إذا كان العميل يرفع بنفسه: استخدم **Satis** لتعطيه حساباً خاصاً به يمكنك إيقافه في أي وقت.
